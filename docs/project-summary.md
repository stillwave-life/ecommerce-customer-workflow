# ecommerce-customer-workflow 项目总结

## 项目定位

`ecommerce-customer-workflow` 当前是一个面向京东客服场景的 OpenClaw 离线 workflow skill MVP。

它不是全自动客服机器人，也不是直接操作京东后台的执行器。当前版本的目标，是先把客服代理最基础、最关键的底层链路做通：

1. 接收不同来源的客服输入
2. 统一整理成结构化上下文
3. 命中本地商品、FAQ、规则知识
4. 生成可审核的中文客服回复草稿

当前项目更准确的定位是：

> 一个可插入 OpenClaw skills 目录、可独立运行、可继续扩展为京东客服代理系统的离线 MVP 底座。

## 当前已经实现的能力

### 1. 标准 OpenClaw skill 包结构

当前项目已经具备完整 skill 包形态，核心结构包括：

- `SKILL.md`
- `_meta.json`
- `README.md`
- `config/default.json`
- `scripts/`
- `app/`
- `references/`
- `examples/`
- `shops/`
- `assets/`
- `templates/`

可以直接作为一个独立目录放入 OpenClaw 的 `skills` 目录中进行测试。

### 2. 三类输入统一处理

当前统一入口是：

- `scripts/prepare_request.py`

已支持三类输入：

- `text`：纯文本输入
- `url`：京东商品 URL
- `image`：本地图片路径

这些输入最终会统一转换为一个标准的 `prepared` 结构，核心字段包括：

- `shop_id`
- `session_id`
- `source_type`
- `source_value`
- `product_ref`
- `user_message`
- `page_context`
- `parsed_entities`
- `knowledge_hits`

### 3. 最小请求校验入口

原始最小校验入口仍然保留：

- `scripts/validate_request.py`

该脚本用于兼容原本的最小输入契约，不破坏旧行为。

### 4. 京东商品 URL 最小识别

当前实现文件：

- `app/parsers/url_parser.py`

当前能力：

- 支持最小京东商品 URL 识别
- 可从 URL 中提取商品 ID
- 将提取结果写入 `product_ref`
- 将 URL、host、path 等写入 `page_context`

当前边界：

- 不抓取网页内容
- 不读取登录态
- 不做反爬
- 不查询实时价格/库存
- 不处理后台页面会话上下文

### 5. 本地图片路径输入支持

当前实现文件：

- `app/parsers/image_parser.py`

当前能力：

- 校验图片路径是否存在
- 输出文件级结构化上下文
- 预留 OCR / 图片识别扩展位
- 在无法识别时保持保守，不编造商品信息

当前边界：

- 尚未完成真正的 OCR
- 尚未完成商品截图识别
- 尚未完成聊天截图识别
- 尚未完成型号/颜色/规格提取

### 6. 本地知识补齐

当前实现文件：

- `app/data/catalog_loader.py`
- `app/data/knowledge_loader.py`

默认数据源：

- `shops/default/products.csv`
- `shops/default/faq.md`
- `shops/default/rules.md`

当前能力：

- 读取商品资料 CSV
- 读取 FAQ 文本
- 读取规则文本
- 根据 `product_ref` 或候选实体做简单命中
- 生成结构化 `knowledge_hits`

### 7. 客服回复草稿生成

当前实现文件：

- `scripts/generate_reply.py`
- `app/reply/reply_builder.py`

当前能力：

- 基于 `prepared` 结果生成中文客服回复草稿
- 输出 `reply_draft`
- 输出 `reply_reasoning_summary`
- 输出 `facts_used`

当前策略：

- 规则模板优先
- 缺少关键事实时输出保守回复
- 不编造库存
- 不编造价格
- 不编造发货承诺
- 不编造平台规则
- 不编造售后承诺

### 8. GitHub 仓库与基础发布文件已完成

当前已经补齐并推送：

- `README.md`
- `.gitignore`
- `LICENSE`

GitHub 仓库：

- `https://github.com/stillwave-life/ecommerce-customer-workflow`

## 项目最终完整形态设想

如果后续继续按既定方向做完整，这个项目最终会成为一个分层的京东客服代理系统。

### 第一层：输入识别层

负责把输入信号转成结构化信息：

- 文本识别
- 京东 URL 识别
- OCR
- 商品图片理解
- 聊天截图理解
- 型号 / 规格 / 颜色 / 数量抽取

### 第二层：知识检索层（RAG）

负责把商品和客服知识真正召回出来：

- 商品资料检索
- FAQ 检索
- 售后规则检索
- 发货规则检索
- 平台规则检索
- 向量召回 + 规则召回混合

未来能力会包括：

- 文档切块
- embedding 建库
- 召回排序
- 低置信度人工确认
- 引用来源标注

### 第三层：回复生成层

负责把已命中的事实组织成真正可用的客服回复：

- 模板回复
- 模型增强回复
- 风险校验
- 禁止违规承诺
- 输出 facts_used
- 输出 reasoning summary

### 第四层：人工审核层

该层是系统的重要安全边界。不是让系统直接自动发，而是先展示：

- `prepared`
- `knowledge_hits`
- `facts_used`
- `reply_draft`

人工可以：

- 审核
- 修改
- 确认发送
- 拒绝执行

### 第五层：执行层

这是后续浏览器自动化或后台插件接入的层。

#### 路线 A：浏览器自动化

- Playwright / 浏览器控制
- 读取已登录京东客服页面
- 定位当前会话
- 自动填充回复框
- 默认不发送
- 显式确认后发送

#### 路线 B：后台插件 / 页面扩展

- 挂载在客服后台页面中
- 展示商品识别结果
- 展示知识命中结果
- 展示候选回复
- 一键填充
- 人工确认发送

相比纯浏览器脚本，插件路线通常更稳定、更适合长期产品化。

## 推荐的后续开发顺序

### 第一步：先做 OCR / 图片识别

原因：

- 这是当前最明显的能力缺口
- 客服场景中截图输入价值很高
- 技术风险低于浏览器自动化

目标：

- 本地图片路径 → OCR 文本
- 聊天截图 → 商品 / 规格 / 颜色 / 型号实体提取
- 商品截图 → 标题 / 型号候选抽取

### 第二步：做 RAG

原因：

- 当前知识命中还只是简单匹配，准确率和召回能力有限
- 真正想让回复生成更稳定，RAG 是必须的

目标：

- FAQ / rules / 商品说明入库
- embedding / 向量检索
- 检索结果排序
- 引用来源输出
- 低置信度命中人工确认

### 第三步：做回复增强

原因：

- 先有识别和检索，再做模型生成才有价值

目标：

- 规则 + 模型混合生成
- grounding 到 facts
- 风险词检测
- 承诺类回复拦截
- 风格控制

### 第四步：做浏览器半自动执行

原因：

- 风险最高
- 页面变化多
- 但业务价值也最大

目标：

- 读取已登录客服后台当前会话
- 自动填充回复框
- 默认不发送
- 显式确认后发送

### 第五步：做后台插件

如果项目要长期稳定使用，后台插件通常会比纯脚本更像产品。

目标：

- 页面内辅助面板
- 商品识别结果展示
- 知识命中展示
- 回复候选展示
- 一键填充 / 二次确认发送
