# Windows UIA 稳定底座方向设计

## 1. 文档目的

本文档用于定义 `ecommerce-customer-workflow` 下一阶段的 Windows UIA 稳定底座方案。目标不是立刻完成真实自动填充，而是优先建立一个低调试成本、可观测、可诊断、可回退的真实 Windows UI Automation 底层能力层，为后续安全接入真实填充与人工发送流程打基础。

本文档解决的是方向、模块职责、诊断策略、动作闸门与阶段边界，不展开逐步编码细节。

---

## 2. 为什么先做稳定底座

如果现在直接尝试“找到输入框并写入文本”，未来最容易出现的问题不是功能缺失，而是：

- 不知道失败发生在哪一层
- 不知道是窗口识别错、节点读取错、输入框定位错，还是聚焦/写入失败
- 一旦真实后台界面有变化，很难定位回归原因
- 写入动作过早接入，风险会明显高于当前阶段的控制能力

因此当前更合理的路线是：

> 先做可观测、可诊断的 UIA 稳定底座，再接真实填充。

---

## 3. 当前阶段目标

这一阶段的目标不是“自动回复”，而是建立一条完整且可诊断的真实 UIA 底层链路，使系统能够：

1. 读取真实前台窗口
2. 枚举真实 UIA 节点
3. 标准化节点结构
4. 统计节点特征与候选输入框
5. 对输入框候选做评分
6. 在动作前执行闸门判断
7. 尝试聚焦输入框
8. 输出完整诊断报告

当前阶段明确不做：

- 真实填充文本
- 自动点击发送
- 自动批量会话处理
- 仅靠坐标点选绕过结构判断

---

## 4. 总体原则

### 4.1 可观测优先

每个关键阶段都必须输出结构化诊断信息，不能只返回“成功/失败”。

### 4.2 动作必须有闸门

真实动作不能建立在低置信度判断上。没有足够置信度，就不应执行 focus 或写入。

### 4.3 读取与动作分层

读取链路、候选评分链路、动作链路必须分开。这样后续填充失败时，可以快速判断问题是在识别层还是动作层。

### 4.4 失败要可解释

失败结果必须带标准化原因，例如：

- `foreground_window_not_found`
- `uia_tree_empty`
- `no_input_candidates`
- `candidate_score_too_low`
- `focus_api_failed`
- `focus_verification_failed`

---

## 5. 推荐模块拆分

### 5.1 `windows_uia_probe`

职责：

- 获取当前前台窗口
- 读取窗口标题、句柄、进程名、bounds
- 拉取 UIA 节点原始信息
- 输出基础探针报告

它只负责“采样与探测”，不负责业务判断。

---

### 5.2 `windows_node_normalizer`

职责：

- 将真实 UIA 节点转换为统一标准结构
- 统一字段命名与布尔属性
- 输出可供 perception pipeline 与评分器消费的节点列表

这一层是未来替换后端实现时最重要的稳定边界。

---

### 5.3 `input_candidate_ranker`

职责：

- 从标准化节点中找出候选输入框
- 对每个候选进行评分
- 输出候选列表与评分原因

推荐评分因素：

- 是否可编辑
- 是否位于窗口下半区域
- 是否为大文本输入区域
- 是否接近发送按钮区域
- 是否不是搜索框
- 是否名称/控制类型匹配输入框特征

它不负责真正执行 focus。

---

### 5.4 `action_gate`

职责：

- 根据候选评分、窗口状态、可见性和编辑性判断是否允许动作
- 明确输出：
  - `allow_focus`
  - `manual_review`
  - `block`

这一层是安全边界，不允许被跳过。

---

### 5.5 `focus_executor`

职责：

- 在 `action_gate` 放行后对目标输入框执行 focus
- 返回焦点结果
- 执行焦点后的验证

此时仍不写入文本。

---

### 5.6 `uia_diagnostics_report`

职责：

- 汇总窗口探测、节点统计、候选评分、闸门判断、焦点结果
- 生成面向开发调试的 JSON 报告

这个报告未来会极大降低 debug 成本。

---

## 6. 推荐数据流

```text
真实前台窗口
  -> windows_uia_probe
  -> windows_node_normalizer
  -> input_candidate_ranker
  -> action_gate
  -> focus_executor
  -> uia_diagnostics_report

同时：
  -> region_classifier
  -> jd_workspace_parser
  -> desktop_context
```

这里有两条并行链：

### 6.1 感知链
用于产出：
- `desktop_context`

### 6.2 动作诊断链
用于产出：
- `candidate_scores`
- `gate_decision`
- `focus_result`
- `diagnostics_report`

感知链和动作诊断链必须分开，但共用标准化节点。

---

## 7. 关键输出结构建议

### 7.1 节点统计输出

```json
{
  "node_count": 423,
  "editable_node_count": 3,
  "clickable_node_count": 27,
  "visible_node_count": 390
}
```

### 7.2 输入框候选评分输出

```json
{
  "candidates": [
    {
      "node": {},
      "score": 0.86,
      "reasons": [
        "editable",
        "bottom_region",
        "large_text_area",
        "near_send_button"
      ]
    }
  ]
}
```

### 7.3 动作闸门输出

```json
{
  "decision": "allow_focus",
  "score": 0.86,
  "reason": "candidate_score_high_enough"
}
```

### 7.4 焦点结果输出

```json
{
  "ok": true,
  "focused": true,
  "target_role": "edit",
  "target_name": "输入框",
  "reason": ""
}
```

### 7.5 诊断报告输出

```json
{
  "window": {},
  "node_stats": {},
  "input_candidates": [],
  "gate_result": {},
  "focus_result": {},
  "desktop_context": {}
}
```

## 8. 当前阶段与后续阶段的关系

### 当前阶段做什么

- 找到真实前台窗口
- 读取真实 UIA 节点
- 识别输入框候选
- 给候选评分
- 判断是否允许 focus
- 执行 focus
- 输出诊断结果

### 当前阶段不做什么

- 不写入真实文本
- 不把 `reply_draft` 直接打进输入框
- 不点击发送
- 不自动切换会话

### 下一阶段才做什么

- 写入测试文本
- 写入后回读校验
- `reply_draft` 到真实填充
- 人工确认发送前的最终联调

---

## 9. 为什么这样更稳定

因为未来一旦失败，就可以从诊断结果直接看出问题在哪：

- 是找不到前台窗口
- 是 UIA 树为空
- 是没有 editable node
- 是输入框候选评分太低
- 是闸门阻断了动作
- 是 focus API 执行失败
- 是 focus 成功但验证失败

这比“填充失败”这种单一报错更可维护得多。

---

## 10. 推荐测试策略

### 10.1 探针单元测试

测试前台窗口、节点标准化、统计结果结构。

### 10.2 候选评分测试

测试不同类型节点在候选评分中是否得到预期排序。

### 10.3 闸门测试

测试高置信度、低置信度、无候选等情况下的动作决策。

### 10.4 焦点结果测试

测试 focus 成功/失败合同。

### 10.5 诊断报告测试

测试报告是否完整包含窗口、候选、闸门、焦点与 `desktop_context`。

---

## 实现入口

UIA 稳定底座第一版从结构化 probe 输入、标准化节点、候选评分、动作闸门、焦点验证与诊断报告合同开始，不直接执行真实文本填充。

这样可以让底座层先具备高可观测性与高可测试性，同时保留长期架构目标：未来真实 Windows UIA backend 与真实写入能力可以建立在相同的诊断、评分、闸门和验证层之上，而不需要推翻现有底座。

---

## 11. 结论

下一阶段应按以下方向推进：

> 先建立一个可观测、可诊断、可回退的真实 Windows UIA 稳定底座，完成真实窗口探测、节点标准化、输入框候选评分、动作闸门与焦点验证，再在此基础上进入真实填充阶段。

这条路线虽然比直接写填充稍慢，但会显著降低未来真实桌面接入阶段的 debug 成本，是更稳的长期路线。
