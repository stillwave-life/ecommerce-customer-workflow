# 京东客服桌面代理方向方案

## 1. 文档目的

本文档用于定义 `ecommerce-customer-workflow` 从当前离线 OpenClaw workflow skill，演进为可在真实京东客服后台环境中辅助人工操作的桌面客服代理的方向方案。

本方案先确定目标形态、阶段划分、系统边界、模块职责与实施顺序。在正式开发前，所有后续实现应以本方向方案为上层约束。

---

## 2. 目标形态

目标不是做一个单纯的离线回复生成脚本，而是做一个可以被 OpenClaw skills 生态调度、并能在真实客服桌面工作流中协助操作的京东客服桌面代理。

当用户说出：

> 京东客服启动

系统应具备以下能力：

1. 识别当前桌面是否处于京东客服工作台
2. 扫描当前界面并识别关键工作区域
3. 理解当前客服会话上下文
4. 结合商品、用户、订单、FAQ、规则与会话记忆生成候选回复
5. 自动将候选回复填充到输入框
6. 将发送动作保留给人工处理

当前阶段不以“完全自动发送”为目标，而以“真实可用、可控、可审查、可逐步增强”的桌面辅助代理为目标。

---

## 3. 当前项目基础

当前仓库已经具备以下可复用基础：

- 离线 skill 包结构
- 标准化请求准备入口
- 本地知识补齐能力
- 保守回复草稿生成能力
- 约束与会话记忆方向的演进基础

当前已有主链路：

- `scripts/validate_request.py`
- `scripts/prepare_request.py`
- `scripts/generate_reply.py`
- `app/data/*`
- `app/constraints/*`
- `app/reply/*`

这意味着新阶段不应推翻现有 workflow，而应新增“桌面感知层”和“动作执行层”，并将当前仓库保留为业务理解与回复生成核心。

---

## 4. 方向判断

### 4.1 推荐方向

推荐采用以下路线：

> OpenClaw skill 生态优先 + 桌面辅助闭环优先

也就是：

- 架构上按 skill 生态拆分
- 落地上优先实现“识别 → 理解 → 生成 → 填充”闭环
- 发送动作保留人工执行

这条路径既能尽快做出真实可用结果，又不会把项目做成一次性桌面脚本。

### 4.2 不推荐方向

当前不建议直接以“识别 → 生成 → 自动发送”的全自动路线作为第一阶段目标，原因包括：

- 一旦误识别会话或商品，发送后不可逆
- 当前项目尚未具备成熟的界面识别稳定性
- 尚未完成完整风险校验与低置信度阻断机制
- 尚未建立操作审计、二次确认与失败回滚机制

因此第一阶段必须停在“自动填充，人工发送”。

---

## 5. 第一阶段正式目标

第一阶段的正式目标定义为：

> 构建一个可由 OpenClaw 触发的京东客服桌面助手，当用户说“京东客服启动”时，系统能够识别当前京东客服工作台、解析当前聊天内容、商品信息、用户/订单信息，调用现有知识与回复生成能力，生成候选回复并自动填入输入框，等待人工确认发送。

这个目标是当前阶段的唯一主目标。

---

## 6. 基于界面截图确认的区域结构

根据现有京东客服后台截图，界面可以稳定拆成以下区域：

### 6.1 左侧会话列表区

职责：

- 展示待处理会话列表
- 展示当前选中的客户会话
- 提供会话切换入口

可提取信息：

- 当前选中客户
- 待处理会话数量
- 会话摘要
- 最近时间

### 6.2 中间主聊天区

职责：

- 展示当前客户的对话内容
- 展示文本、图片及已读状态
- 展示当前客户标识

可提取信息：

- 最近一条客户消息
- 最近若干轮对话
- 消息类型（文本 / 图片 / 混合）
- 当前客户 ID

### 6.3 底部输入区

职责：

- 承载智能回复建议
- 承载人工编辑与发送入口
- 承载真正的回复输入框

可执行动作：

- 聚焦输入框
- 自动填入文本
- 回读校验内容是否一致
- 停在发送前

### 6.4 中右辅助分类区

职责：

- 展示售前咨询、邀评语、保修政策、快递查询等辅助分类与短语入口

作用判断：

- 可作为后续问题分类与回复策略增强的辅助来源
- 第一阶段不将其作为主链路硬依赖

### 6.5 最右上下文面板区

职责：

- 在不同 tab 下展示商品、用户、订单、服务单等上下文信息

当前已见状态包括：

- 首页
- 服务单
- 商品
- 用户信息
- 订单信息
- 推荐商品列表

结论：

- 最右面板必须被识别为“多状态上下文面板”
- 不能写成单一模板逻辑
- 第一阶段应支持识别聊天区 + 商品区 + 用户/订单区

## 7. 系统分层设计

为了避免未来 UI 变化导致整个系统失效，系统必须分层。

### 7.1 UI 感知层

职责：

- 判断当前是否为京东客服工作台
- 对窗口截图或 UI 结构进行区域切分
- 定位聊天区、商品区、用户订单区、输入框、发送按钮
- 提取界面文本与结构化控件信息

这一层只回答“看到了什么”，不回答“业务上该怎么回复”。

### 7.2 业务理解层

职责：

- 从聊天区提取当前客户问题
- 从商品区提取候选商品与 SKU
- 从用户/订单区提取订单与身份上下文
- 判断问题类型与当前回复目标
- 标记不确定字段与低置信度项

这一层负责把 UI 信息转成标准客服上下文。

### 7.3 知识与回复层

职责：

- 将结构化客服上下文映射到现有 `prepared` 结构
- 命中本地商品、FAQ、规则、会话记忆、约束
- 基于事实生成保守回复草稿
- 输出 `reply_draft`、`facts_used`、`reply_reasoning_summary`

这一层将继续复用当前仓库已有核心能力。

### 7.4 页面动作层

职责：

- 聚焦输入框
- 自动填入回复文本
- 回读校验写入结果
- 返回“已填充，等待人工发送”状态

这一层不负责自动发送。

---

## 8. 第一阶段 skill 生态拆分建议

第一阶段不建议做成一个庞大的单 skill，而应最少拆成以下 4 个核心 skill：

### 8.1 `jd-cs-launch`

职责：

- 响应“京东客服启动”
- 建立当前客服代理 session
- 判断当前窗口是否符合京东客服工作台特征
- 调度后续识别、理解、回复与填充流程

### 8.2 `jd-cs-screen-parse`

职责：

- 截取当前工作台界面
- 切分左侧会话区、中间聊天区、右侧上下文区、底部输入区
- 提取聊天文本、图片存在性、商品信息、用户信息、订单摘要
- 输出统一的 screen context

### 8.3 `jd-cs-reply-compose`

职责：

- 将 screen context 转为标准 `prepared` 结构
- 调用本地知识、规则、记忆、约束与回复生成逻辑
- 输出最终候选回复与依据

### 8.4 `jd-cs-fill-input`

职责：

- 定位底部输入框
- 自动填入候选回复
- 对输入框内容进行回读校验
- 返回人工发送等待状态

## 9. 第一阶段识别范围

根据你的要求，第一阶段不是只识别聊天区，而是识别完整上下文。

第一阶段必须覆盖：

### 9.1 聊天区识别

必须提取：

- 当前客户 ID
- 最近一条客户消息
- 最近若干轮对话摘要
- 是否包含图片或混合输入
- 当前消息是否更像售前、售后、物流、规格、库存、价格、邀请评价等问题

### 9.2 商品区识别

必须提取：

- 商品标题
- SKU / 商品编码
- 价格
- 库存状态
- 规格候选
- 当前右侧商品 tab 状态

### 9.3 用户 / 订单区识别

必须提取：

- 客户昵称/标识
- 用户等级或标签
- 近三个月订单或历史订单摘要（若当前界面可见）
- 服务单/订单搜索状态
- 当前面板是否可提供订单上下文

### 9.4 输入区识别

必须提取：

- 输入框位置
- 当前是否可编辑
- 当前已有内容是否为空
- 发送按钮是否可见
- 智能回复建议是否存在

## 10. 输入输出协议方向

### 10.1 启动输入

建议启动协议：

```json
{
  "command": "京东客服启动",
  "mode": "assist_fill"
}
```

### 10.2 screen parse 输出

建议标准结构：

```json
{
  "ok": true,
  "platform": "jd_customer_service",
  "confidence": 0.0,
  "active_customer": {
    "id": "",
    "name": ""
  },
  "chat_context": {
    "latest_customer_message": "",
    "recent_messages": [],
    "contains_image": false
  },
  "product_context": {
    "tab_active": false,
    "items": []
  },
  "user_order_context": {
    "user_labels": [],
    "orders": [],
    "service_forms": []
  },
  "input_context": {
    "editable": false,
    "has_smart_reply": false,
    "send_button_visible": false
  }
}
```

### 10.3 compose 输出

继续对接现有风格：

```json
{
  "ok": true,
  "prepared": {},
  "reply_draft": "",
  "reply_reasoning_summary": "",
  "facts_used": []
}
```

### 10.4 fill 输出

```json
{
  "ok": true,
  "filled": true,
  "verified": true,
  "ready_for_manual_send": true
}
```

## 11. 技术策略原则

### 11.1 识别后端策略

不能一开始就把系统做成纯截图坐标脚本。

推荐采用：

> UI Automation / Accessibility 优先 + OCR / screenshot 兜底 + 坐标点击作为最后手段

原因：

- 纯坐标点击脆弱
- 纯 OCR 对复杂控件不稳定
- 如果可访问控件树可用，将显著提升稳定性
- 截图识别应作为跨环境兜底方案

### 11.2 回复生成策略

必须继续坚持当前项目的保守策略：

- 不编造库存
- 不编造价格
- 不编造发货承诺
- 不编造售后承诺
- 信息不足时优先澄清

### 11.3 动作执行策略

第一阶段只允许：

- 聚焦输入框
- 自动填入文本
- 回读核对
- 停止在发送前

禁止：

- 自动点击发送
- 自动切换大量会话
- 自动批量回复

## 12. 阶段路线图

### Phase A：桌面客服助手 MVP

目标：

- 完成“京东客服启动”
- 完成当前界面识别
- 完成聊天区 + 商品区 + 用户/订单区识别
- 完成候选回复生成
- 完成自动填充输入框
- 人工手动发送

### Phase B：稳定化增强

目标：

- 多轮会话理解
- 图片/OCR 增强
- 商品候选排序
- 低置信度阻断
- 风险词检测
- 输入区写入后回读校验
- 右侧上下文面板多状态适配

### Phase C：受控发送能力

目标：

- 显式确认发送
- 发送前二次校验
- 操作审计
- 白名单低风险场景自动发送

Phase C 不属于当前阶段目标。

---

## 13. 当前阶段不做事项

当前方向明确不做：

- 第一阶段自动点击发送
- 风控绕过
- 京东后台账号接管逻辑
- 大规模批量自动回复
- 以实时联网抓取作为主依赖
- 把 UI 感知、业务理解、回复生成、动作执行耦合进单一模块

---

## 14. 下一步执行建议

在进入实现前，应先基于本方向方案输出一份更细的实施计划，至少包括：

1. 第一阶段 skill 目录与模块边界
2. screen context 标准结构
3. UI 感知层的候选技术路线
4. 从截图识别结果映射到 `prepared` 的协议
5. 输入框填充与回读校验流程
6. 第一阶段验证方法

该实施计划应作为后续编码工作的直接依据。

---

## Phase A Implementation Contracts

Phase A exposes one script entrypoint:

- `scripts/jd_customer_service_start.py`

The entrypoint accepts a JSON payload containing:

- `command`: must be `京东客服启动`
- `shop_id`: shop identifier
- `session_id`: desktop assistant session identifier
- `desktop_context`: parsed screen context

Phase A does not capture the screen directly. It accepts already parsed `desktop_context` so UI Automation, OCR, and screenshot parsers can be added behind the same contract later.

Phase A output may include a `fill_action`, but `fill_action.auto_send_allowed` must always be `false`.

---

## 15. 结论

本项目下一步不是继续扩展单纯离线回复逻辑，而是升级为：

> 一个可在 OpenClaw 中被口令触发、能够理解京东客服桌面工作台、整合聊天/商品/用户/订单上下文、生成保守客服回复并自动填充输入框的人机协作式桌面客服代理。

当前第一阶段唯一正确目标是：

> 识别 + 理解 + 生成 + 填充，发送由人工完成。
